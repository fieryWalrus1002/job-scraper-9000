# Frontend and API Structure

Alright, now we start laying out the groundwork for our frontend and API. We've got an okayish working pipeline that scrapes jobs, filters them, and scores them. But how do we actually interact with this data? That's where the frontend and API come in.

The idea is to have a clean separation between the backend (FastAPI) and the frontend (React + Vite). The backend will handle all the data processing, job scoring, and serve that data through a RESTful API. The frontend will consume this API and provide a user-friendly interface for users to interact with the job data.

We have used Python & FastAPI for the backend because thats where we're best. For the frontend, I want to use TypeScript + React with Vite for a modern, fast development experience. We'll also use shadcn/ui for pre-built components like tables and cards to display the job listings and their scores.

> **UI library status (updated 2026-06-01):** The MVP was built with plain CSS and a plain HTML table — no Tailwind, no shadcn/ui, no TanStack Table, no Recharts. The component count didn't justify the toolchain overhead at that scale. Revisit shadcn/ui when the app grows beyond a single table + summary view, or when consistent component styling across multiple pages becomes a maintenance burden.

Frontend directory will have its own package.json. All the Node.js tools and dependencies for the frontend will be contained there. Python dependencies will be in the same root pyproject.toml.

## Local Development

For local dev, we'll be using a uvicorn server to run the FastAPI backend and Vite's dev server for the frontend. The Justfile will have commands to start both servers, and we can set up CORS in FastAPI to allow the frontend to make API requests during development.

### Justfile Recipes

We will use `just` to manage running the full stack locally without needing to manage multiple terminal paths manually. I've already added these:

```just
# Start the FastAPI backend server
backend:
    uvicorn src.api.main:app --reload --port 8000

# Start the Vite frontend development server
frontend:
    cd frontend && npm run dev -- --port 5173

# Run both backend and frontend concurrently (requires parallel execution or separate tabs)
dev:
    just backend & just frontend
```

### Vite Proxy (primary) and CORS (fallback)

In local dev, Vite's `server.proxy` forwards `/api/*` to `http://localhost:8000`. Frontend code uses same-origin relative paths (`/api/jobs`), which mirrors the Azure SWA production routing. FastAPI CORS headers are kept as a fallback for direct API access (curl, Postman) but are not the primary development path.

## Migration to Azure Cloud

For production deployment, we will utilize a modern, serverless cloud architecture on Azure that optimizes cost and separation of concerns:

```Plaintext
[ Browser ] ───> [ Azure Static Web Apps (Frontend) ]
│
└─── (Routed API Traffic) ───> [ Azure Container Apps (FastAPI Backend) ]
```

### 1. Frontend: Azure Static Web Apps (SWA)

- **Why:** SWA is tailor-made for static application builds generated by tools like Vite. It serves assets directly from an Azure CDN for high performance and near-zero cost.
- **CI/CD:** It integrates natively with GitHub Actions; pushing to the `main` branch will automatically trigger Vite to compile the production build and deploy it. THE LESS I NEED TO REMEMBER TO DO MANUALLY THE BETTER.

### 2. Backend & Ingestion: Azure Container Apps (ACA)

We will split our backend into two distinct execution environments inside Azure Container Apps to isolate our ingestion workload from our web traffic:

- **ACA App (FastAPI Backend):** A reactive, web-facing container configured to scale to zero. It handles client requests, authentication, and database reads for the dashboard.
- **ACA Job (Scraper & Ingestion Pipeline):** A scheduled, headless container that spins up automatically via a cron trigger at midnight. It runs the automated scraping, processing, and LLM scoring pipeline to completion, writes the data to persistent storage, and terminates. We're already effectively doing this with `at 12:30 AM -f scripts/run_overnight.sh` locally, so this is just a cloud migration of that existing workflow.

### 3. Eliminating Production CORS

Azure Static Web Apps allows us to configure a `staticwebapp.config.json` file to route traffic sent to `/api/*` directly to our backend Container App. This allows the frontend to talk to the backend on the *same domain* in production, completely eliminating the need to expose open CORS headers in the production API.

### 4. Database architecture

Two-schema Postgres approach, chosen over Azure Blob Storage for the pipeline data:

- **`raw` schema** — append-only landing zone for scored job postings. DDL owned by `scripts/db_ingest.py` (`db/schema.sql`). dbt treats these tables as read-only Sources. No Alembic migrations here — the ingest script applies DDL directly and rebuilds are safe.
- **`app` schema** — user application state (tracked jobs, notes, status). Owned by Alembic + SQLModel. Safe, incremental migrations. Never touched during pipeline rebuilds.

In Azure: the ACA Job writes to `raw` via the ingest script. The ACA App (FastAPI) reads `raw` and reads/writes `app`. Both point at the same Azure Database for PostgreSQL instance.
